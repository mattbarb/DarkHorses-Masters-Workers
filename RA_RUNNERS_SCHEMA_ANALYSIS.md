## üîë RUNNER_ID ANALYSIS

**User's Question:** "Runner_id is no needed as the data is actually a merge of race_id and horse_id"

### Current Implementation
```python
# fetchers/races_fetcher.py line 269
runner_id = f"{race_id}_{horse_id}"
```

### Analysis

‚úÖ **runner_id IS valid** - It's a **surrogate key** pattern, not redundant data:

| Approach | Pros | Cons |
|----------|------|------|
| **Current: runner_id** (single PK) | ‚Ä¢ Simple queries<br>‚Ä¢ Standard single-column PK<br>‚Ä¢ Easy foreign key references<br>‚Ä¢ Common pattern in DB design | ‚Ä¢ Derived from race_id + horse_id<br>‚Ä¢ Additional column |
| **Alternative: (race_id, horse_id)** (composite PK) | ‚Ä¢ No extra column<br>‚Ä¢ "Pure" relational design | ‚Ä¢ Complex JOIN queries<br>‚Ä¢ Harder foreign key references<br>‚Ä¢ More verbose SQL |

### Recommendation

**KEEP runner_id AS IS** ‚úÖ

This is a valid and widely-used database design pattern called a **surrogate key**. While technically derived from race_id + horse_id, it provides:
1. Simple single-column PRIMARY KEY
2. Easier query syntax
3. Cleaner foreign key relationships
4. Industry-standard approach

The alternative (composite PK) is academically "purer" but creates practical query complexity.
